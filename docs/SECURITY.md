# Security - TMF Coders Infrastructure

Bank-grade security posture for the Scaleway multi-tenant infrastructure.

## Controls

| Domain | Control |
|--------|---------|
| Tenant isolation | One Project (or Organization) per tenant; no shared VPC |
| Network ingress | Default-deny security groups; no public VM IPs |
| Remote access | SSH only via the Public Gateway bastion (port 61000) |
| Public app exposure | Odoo reachable only through the Load Balancer (HTTPS) |
| Identity | One least-privilege IAM application per workload; no personal keys |
| CI identity | Dedicated Org-wide IAM application + key, stored as GitHub secrets |
| Secrets | Generated (`random_password`), stored in Secret Manager, pulled at boot |
| State | Private, versioned Object Storage bucket; `use_lockfile` state locking |
| State keys | Tenant-scoped (`tenants/<t>/<layer>/<env>/...`) — no cross-tenant reads |
| Audit immutability | Object Lock COMPLIANCE mode, 730 days, on the audit bucket |
| Data residency | EU regions only, enforced by `region` variable validation |
| Encryption at rest | Scaleway block storage and Object Storage encrypt by default |
| Encryption in transit | LB TLS (Let's Encrypt); bastion SSH |

## Secrets handling

- No secret value is ever written to a `.tfvars` file.
- Passwords are generated by `random_password` and stored in Scaleway Secret
  Manager (`1-org` for the Odoo master password, `3-apps` for the DB password).
- VMs pull secrets at boot via `scw secret version access`, authenticating with
  a project-scoped workload IAM key.
- The workload key reaches VM cloud-init through `user_data`; it is the minimal
  credential (Secret Manager read on one project) and is rotatable.
- Tenant `.tfvars` are committed because they carry no secrets — only the
  tenant inventory. `gitleaks` scans every commit and PR as a backstop.

## Threat model (summary)

| Threat | Mitigation |
|--------|------------|
| Internet scan / brute force on VMs | No public IPs; bastion-only SSH |
| Lateral movement between tenants | Per-tenant Project/VPC; project-scoped IAM |
| Stolen CI key | Org-scoped but rotatable; GitHub secret scoping; audit trail |
| Stolen workload key | Reads one project's secrets only; rotate via `1-org` |
| State file exfiltration | Private bucket, IAM-scoped; secrets also in Secret Manager |
| Tampering with audit logs | Object Lock COMPLIANCE — immutable until expiry |
| Accidental destroy | State versioning; RDB backups; Object Lock on audit data |

## Hardening checklist

- [ ] `state_bucket_suffix` is unique and the bucket is private.
- [ ] CI keys stored as GitHub secrets, never committed.
- [ ] `odoo_domain` set so the LB serves HTTPS (not plain HTTP).
- [ ] `rdb_is_ha = true` in production.
- [ ] Workload and CI keys rotated quarterly.
- [ ] `make check` green (fmt, validate, tflint, tfsec); CI gate enforced.
- [ ] Bastion access restricted to known operator IPs at the org level.

## CI security gate

Every PR runs: `terraform fmt`/`validate`, `tflint`, `tfsec` (fail on HIGH+),
`checkov` (SARIF upload, fail on HIGH+), `gitleaks` (secret scan),
`terraform-docs` drift check, `infracost` cost diff.
